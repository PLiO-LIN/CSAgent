# CSAgent 迁移策略

## 1. 迁移目标

迁移不是把 `telecom-agent` 全量原样拷贝到新目录，而是把现有项目中的能力拆成三层：

- 框架核心
- 插件标准
- 电信业务实现

## 2. 当前迁移原则

### 2.1 先稳定框架边界

优先在 `CSAgent` 中稳定以下边界：

- 对话接口边界
- Tool / Card / Skill 协议边界
- 插件目录边界
- 配置边界

### 2.2 先桥接，再搬迁

当前推荐先通过 `legacy_bridge` 保持能力可用，再把旧版能力逐步迁入：

- 查询插件
- 推荐插件
- 下单插件
- 充值插件

### 2.3 先迁规则清晰的能力

建议优先迁移：

- `load_skills`
- `search_knowledge` 的外接协议
- 查询类工具
- Skill Manifest 和 Skill Prompt

建议后迁移：

- 下单确认
- 支付确认
- 工作流状态机
- 复杂卡片闭环

## 3. 模块映射

当前项目到新框架的推荐映射如下：

| 旧目录 | 新目录 |
| --- | --- |
| `backend/routes` | `backend/app/routes` |
| `backend/agent` | `backend/app/runtime` |
| `backend/tool/base.py` | `backend/app/runtime` 或未来 `backend/app/tools` |
| `backend/skill/base.py` | `backend/app/plugins` 或未来 `backend/app/skills` |
| `backend/card` | `frontend/src/components` + `docs/04-插件化开发标准.md` |
| `backend/tool/*.py` | `plugins/*/backend/tools/*.py` |
| `backend/skill/skills/*` | `plugins/*/backend/skills/*` |
| `frontend/src/components/*Card.tsx` | `plugins/*/frontend/cards/*.tsx` 或通用卡片层 |

## 4. 推荐迁移顺序

## 第一阶段：框架承载层

- 建立 `CSAgent` 目录
- 建立后端入口
- 建立前端壳
- 建立插件目录
- 建立标准文档

## 第二阶段：元数据与协议迁移

- 插件 Manifest
- Skill frontmatter 扩展
- Card schema 版本化
- Tool policy 标准化

## 第三阶段：功能迁移

### 3.1 全局插件

- `global.base`
- `global.knowledge`

### 3.2 电信查询插件

- 套餐
- 余额
- 账单
- 积分
- 用户信息

### 3.3 电信推荐插件

- 推荐
- 对比
- 推荐卡片

### 3.4 电信订单插件

- 限制校验
- 预下单
- 验证码
- 提交订单
- 支付确认

### 3.5 电信充值插件

- 充值预览
- 充值链接
- 充值卡片

## 5. 迁移完成标志

当出现以下状态时，可以认为旧版能力已经基本完成迁移：

- `legacy_bridge` 不再承担主要聊天链路
- 原生 `native_core` 可以跑通主链路
- 插件扫描可以自动发现 Tool / Skill / Card 定义
- 前端可以按插件注册表加载卡片渲染器

## 6. 当前建议

你接下来可以直接在 `CSAgent` 里推进以下事情：

1. 优先把插件发现和加载机制做实
2. 再把 Skill 和 Tool 的标准声明做实
3. 再按业务域迁移电信插件
4. 最后去掉对旧仓库的桥接依赖
