## 4. 当前执行建议

当前已经完成一次“先对齐能力、再继续抽象”的迁移。接下来建议按下面方式继续处理：

- 保持现有 `backend` 原生运行时持续可用
- 从查询类能力开始，把后端工具、技能和前端卡片逐步抽回 `plugins/`
- 框架级能力逐步沉淀到 `backend/app`
- 文档、schema、测试与插件一起维护

## 5. 当前最推荐起步点

如果你现在马上要开始下一步实现，我建议先从 `telecom.query` 开始，因为：

- 查询类 Tool 最适合先标准化
- 卡片类型相对稳定
- 对工作流副作用最小
- 能最快验证插件化路线是否正确
