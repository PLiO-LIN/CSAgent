# CSAgent 插件化开发标准

## 1. 核心关系

在 CSAgent 中，Tool、Card、Skill 三者关系固定如下：

- Skill 管 Tool
- Tool 产 Card
- Card 回对话

知识能力单独处理：

- 知识能力是全局工具
- 不属于某个业务 Skill 私有能力
- 应面向外部知识库接入

## 2. 插件目录标准

每个插件目录至少包含：

```text
plugins/<plugin_id>/
  plugin.yaml
  README.md
  backend/
    skills/
      <skill>/SKILL.md
```

如果该插件已经开始真正实现 Tool / Card，则推荐扩展成：

```text
plugins/<plugin_id>/
  plugin.yaml
  README.md
  backend/
    tools/
    skills/
  frontend/
    cards/
  schemas/
  tests/
```

## 3. Manifest 标准

每个插件必须有 `plugin.yaml`。

推荐字段：

- `plugin_id`
- `name`
- `version`
- `sdk_version`
- `kind`
- `enabled`
- `owner`
- `summary`
- `compatibility`
- `exports`
- `dependencies`

### kind 取值建议

- `global_tool`
- `capability`
- 未来可扩展为 `shared_ui`、`connector`

## 4. Tool 标准

### Tool 职责

Tool 只负责：

- 接收结构化参数
- 执行业务能力
- 返回摘要文本和结构化结果
- 按需产出 Card
- 按需回写 workflow state

Tool 不负责：

- 场景编排
- 前端渲染
- 直接决定对话策略

### Tool 输出建议

Tool 统一返回 `ToolResult` 风格结构：

- `text`
- `error`
- `metadata`

如果要返回卡片：

- 单卡：`metadata.card`
- 多卡：`metadata.cards`

如果要更新工作流：

- `metadata.workflow_patch`
- `metadata.workflow_history`

## 5. Card 标准

Card 是结构化 UI 协议对象。

### Card 顶层建议字段

- `type`
- `schema_version`
- `title`
- `summary`
- `status`
- `actions`
- `editableFields`
- `fallbackText`
- `_meta`

### Card 约束

- Card 不直接调后端工具
- Card 不直接调业务接口
- Card 只能把用户动作回流为标准化对话输入

### Action 约束

Card 中的 Action 应表达用户意图，而不是函数调用语法。

推荐：

- `请为我继续生成充值链接`
- `请帮我查询订单状态`

不推荐：

- `调用 create_recharge_link`
- `执行 submit_order`

## 6. Skill 标准

Skill 是场景编排单元。

### Skill 职责

- 声明当前场景可用工具
- 说明使用顺序和约束
- 定义回复风格与退出条件

### Skill 文件形式

Skill 必须使用 `SKILL.md`，包含：

- YAML frontmatter
- Markdown body

### frontmatter 推荐字段

- `name`
- `version`
- `description`
- `tools`
- `card_types`
- `entry_intents`
- `phases`
- `global_tools`

## 7. 全局工具与业务插件边界

以下能力应优先视为全局能力：

- `load_skills`
- `search_knowledge`
- 未来的统一上下文查询能力
- 未来的插件检查能力

业务插件只负责本领域能力，不应重复实现全局能力。

## 8. 版本规则

建议统一采用语义化版本：

- 插件版本
- Skill 版本
- Card schema 版本
- 框架 SDK 版本

如果发生破坏性变更：

- 升 major
- 或生成新的 Card type / Skill 版本

## 9. 开发检查清单

### Tool

- 是否是原子业务能力
- 是否声明稳定参数 schema
- 是否按标准输出结果
- 是否避免直接耦合前端

### Card

- 是否定义 `type`
- 是否定义 `schema_version`
- Action 是否回对话而不是直调接口
- 是否支持可编辑字段和降级文案

### Skill

- 是否是场景而不是功能点
- 是否只负责编排和约束
- 是否明确说明何时可使用全局工具

## 10. 当前项目执行口径

在当前 `CSAgent` 中，建议所有新能力都按下述方式开发：

1. 先建插件目录和 `plugin.yaml`
2. 再写 `SKILL.md`
3. 再实现 Tool
4. 再补前端 Card 组件
5. 最后加测试和示例

这条顺序的好处是：

- 能先稳定能力边界
- 不容易把代码继续写成单体业务项目
- 便于后期脚手架自动生成
