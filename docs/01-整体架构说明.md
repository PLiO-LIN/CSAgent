# CSAgent 整体架构说明

## 1. 目标

CSAgent 的目标不是做一个无边界的通用 Agent 平台，而是做一个适合客服场景长期演进的通用框架。

当前设计重点是把以下能力稳定为框架层：

- 对话接口
- 会话管理
- 插件注册
- Tool / Card / Skill 协作协议
- 前端通用卡片渲染
- 与旧版能力的兼容桥接

## 2. 总体分层

```text
用户 / 第三方接入
  -> frontend
  -> backend/routes
  -> backend/runtime
  -> backend/plugins
  -> domain plugins
  -> external services / legacy backend
```

## 3. 后端分层

### 3.1 routes

`backend/app/routes` 负责 HTTP 接口暴露。

当前保留与旧版一致的接口形状：

- `/api/chat`
- `/api/chat/sse`
- `/api/sessions`
- `/health`

### 3.2 runtime

`backend/app/runtime` 负责聊天运行期逻辑。

当前分为两种模式：

- `legacy_bridge`
- `native_core`

#### legacy_bridge

该模式下，CSAgent 自己负责：

- 新框架入口
- 配置管理
- 插件目录
- 前端承载

但聊天接口和会话接口会代理到旧版 `telecom-agent`。

适合现阶段目的：

- 保留现有功能
- 新目录继续迭代
- 避免一次性大迁移

#### native_core

该模式下面向未来纯框架运行。

当前仅实现了最小本地骨架：

- 原生会话存储
- 原生聊天响应骨架
- 插件扫描结果展示

后续要逐步承接：

- Tool Registry
- Skill Loader
- Card Registry
- Agent Loop
- Workflow State
- Provider Layer

### 3.3 plugins

`backend/app/plugins` 负责读取插件 Manifest。

当前先做这几件事：

- 扫描 `plugins/*/plugin.yaml`
- 建立插件元数据索引
- 提供插件列表给运行时和健康检查使用

后续会继续补：

- 插件依赖校验
- 插件版本兼容
- Tool / Skill 自动发现
- Card Schema 校验

## 4. 前端分层

### 4.1 通用聊天壳

前端不是直接绑电信业务页面，而是先提供通用聊天壳：

- 会话 ID 管理
- 流式与阻塞模式切换
- 文本消息展示
- 卡片渲染和动作回流

### 4.2 通用卡片渲染器

当前前端不内置大量电信专属组件，而是先实现一个通用卡片渲染器：

- 可展示常见标量字段
- 可展示 `items` / `products` 列表
- 可处理 `editableFields`
- 可执行 `actions`

这样可以先承载现有卡片协议，再逐步演进到插件化卡片注册。

## 5. 插件层

当前插件目录按能力域组织：

- `global.base`
- `global.knowledge`
- `telecom.query`
- `telecom.recommend`
- `telecom.order`
- `telecom.recharge`

它们当前主要承担两类作用：

- 固化目录结构和标准化交付形式
- 作为迁移旧版能力的目标落点

## 6. 当前推荐开发方式

在当前阶段，推荐采用下面的方式继续推进：

1. 继续让旧版 `telecom-agent` 提供生产可用能力
2. 让 `CSAgent` 承担框架目录和新标准承载层
3. 逐个把 Tool / Card / Skill 迁移到 `plugins/`
4. 逐步把后端运行期从 `legacy_bridge` 切到 `native_core`

## 7. 为什么先做兼容桥接

如果现在立即把全部代码搬进来，会有两个问题：

- 新框架边界还没稳定，容易边搬边乱
- 当前功能链路太长，整体迁移成本过高

因此当前初版结构采用“先搭框架，再做迁移”的方式更稳妥。

## 8. 当前阶段产物定义

当前 `CSAgent` 应被视为：

- 一个正式的新框架仓库起点
- 一个兼容旧版功能的过渡宿主
- 一个未来承载插件化客服能力的长期目录
